## STRATEGIES AND HARD RULES
[shr-00001] Make sure to end code blocks with ``` followed by a newline(\n).
[shr-00005] Always look at API specifications (using apis.api_docs.show_api_doc) before calling an API.
[shr-00006] Write small chunks of code and only one chunk of code in every step. Make sure everything is working correctly before making any irreversible change.

[shr-00009] Do not infer API names or filesystem method names from generic conventions. In these environments, the exact callable names and parameter names matter (for example, account deletion may be `delete_account`, and file operations may be `create_file`/`update_file`/`show_file` rather than read/write aliases), so verify them in docs before coding.
[shr-00017] Treat destructive actions and task finalization as separate steps. After `delete_account` or similar irreversible calls, do not assume task completion feedback will appear in the same output; if the task must be finalized, call `supervisor.complete_task(...)` explicitly in a separate confirmed step.
[shr-00042] In file-organization tasks where grouping depends on when files were created, fetch per-file metadata with `file_system.show_file` and classify using the authoritative `created_at` value; do not infer groups from filenames, listing order, or assumed date patterns in the directory output.
[shr-00043] For filesystem organization tasks phrased with relative dates such as 'this year', 'this month', or named months, parse each file's `created_at` into year-month components and compare against the task's relative date reference before routing files. Do not classify by month alone when the task scope depends on a specific year; files from other years that share the same month should remain in the fallback group.
[shr-00048] For cross-app media-selection tasks, first extract the concrete requirement from the source app (for example today's workout duration from Simple Note), then choose Spotify content that satisfies that requirement using authoritative duration/track-count data. Do not substitute generic popularity, rating, or like-count heuristics for the task's actual constraint.
[shr-00052] For cross-app Spotify selection tasks with a duration requirement (for example, enough music to cover a workout), extract the required duration from the source app and compare it against authoritative playlist runtime, not just playlist length or song count. A playlist with more tracks can still be semantically wrong or too short/long for the task.
[shr-00053] For filesystem receipt-search tasks, do not assume the receipt is a single PDF or uniquely named file. A substring search may return many monthly bill files; inspect the returned paths, identify the latest relevant bill from the path/filename when the naming is date-like, then read that specific file to extract the amount.
## APIs TO USE FOR SPECIFIC INFORMATION
[api-00004] You can use the "supervisor" app to get information about my accounts and use the "phone" app to get information about friends and family.

[api-00010] For Spotify library exports, use `show_song_library`, `show_album_library`, and `show_playlist_library` to collect the union of library contents. Album and playlist library responses include `song_ids`, while song-library items already include `title` and nested `artists` objects; fetch `show_song(song_id=...)` for deduplicated ids when you need full song metadata for items referenced only by album/playlist membership.
[api-00014] For Spotify playlist-library tasks that require all playlists, inspect whether `show_playlist_library` exposes an `is_public` filter. If it does, fetch both public and private playlists explicitly (e.g. `is_public=True` and `is_public=False`) or otherwise verify that the unfiltered call already includes both, then deduplicate by `playlist_id` before downstream processing.
[api-00018] For relationship-scoped payment tasks (for example coworker/friend/roommate payments), identify the relevant people from the Phone contacts app first, typically via `search_contacts(relationship=...)`, then match Venmo transactions against authoritative contact fields such as email rather than relying on names, descriptions, or social-feed guesses.
[api-00027] For Venmo tasks scoped to a set of known people, call `show_transactions` separately for each authoritative contact email using the `user_email` filter (and the appropriate `direction`, date bounds, and pagination), then union the results by `transaction_id` before taking action. Do not rely on a single broad fetch plus name matching when the API can filter by counterparty email.
[api-00028] For Venmo payment-request tasks where a note lists people by name, use Phone contacts to resolve each intended recipient to an authoritative email, then compare against already-paid Venmo transactions by sender email and send requests with `create_payment_request(user_email=..., amount=..., description=...)`. Do not rely on `venmo.search_users` name search alone to pick recipients, because it can return large ambiguous candidate sets.
[api-00032] In Venmo payment-request tasks, `search_users(query=...)` can return many loosely related candidates. Treat its results as ambiguous by default; disambiguate with authoritative Phone contact email and supporting fields like `friends_since`, or prefer direct email-based APIs when available, rather than selecting the first name match.
[api-00036] Before creating Venmo payment requests, inspect `show_sent_payment_requests(access_token=..., status='pending', page_index=...)` across pages and compare receiver/amount/description to your intended requests so you do not send duplicates when a pending request already exists.
[api-00056] For Phone contact tasks scoped by relationship (for example roommate, coworker, friend, parent), prefer `search_contacts(relationship=...)` after confirming the relationship label, or explicitly filter returned contacts by the `relationships` field. Do not rely on `query='<relationship>'` alone, because text search can return many unrelated contacts whose other fields happen to match the query.

## VERIFICATION CHECKLIST

[vc-00011] When reading API docs, record the full callable contract, not just the endpoint name: required auth fields, optional pagination params like `page_limit`, and the exact response field shapes. In Spotify responses, `artists` may be a list of objects `{id, name}`, and album/playlist detail `songs` may use fields like `id` and `artist_ids` rather than the same shape as `show_song`.
[vc-00012] For file_system writes, verify the documented path/parameter conventions before acting: paths can use `~/...`, `create_directory` expects `directory_path`, `access_token`, and optional `recursive`, and `create_file` expects `file_path`, `access_token`, optional `content`, and optional `overwrite`, returning both a message and `file_path`.
[vc-00015] When using `supervisor.show_account_passwords`, do not expect usernames in the response. It returns a list of `{account_name, password}` entries only; obtain the login email/username from the task context or another authoritative account source before calling app `login`.
[vc-00016] Before any destructive action that depends on a created backup, verify the backup with `file_system.show_file` and inspect the actual returned `content` plus metadata fields such as `file_id`, `path`, `created_at`, and `updated_at` rather than relying on a truncated preview or assumed success message.
[vc-00019] In Venmo, do not guess transaction or social APIs from generic names: use the documented callables such as `show_transactions`, `create_transaction_comment`, and `like_transaction`. `show_transactions` returns objects keyed by `transaction_id`, `created_at`, `sender`, and `receiver` and supports filters including `direction`, `user_email`, `min_created_at`, `max_created_at`, `page_index`, and `page_limit`.
[vc-00020] For Venmo login, the credential field is `username`, not `email`, and successful auth returns both `access_token` and `token_type`; preserve the concrete token for subsequent authenticated calls.
[vc-00021] For Phone contact tasks, verify the exact interfaces: `search_contacts` uses path `/contacts` with optional `query`, `relationship`, `page_index`, and `page_limit`, and returns contact records with fields like `contact_id`, `first_name`, `last_name`, `email`, `phone_number`, and `relationships`.
[vc-00022] Do not assume the Phone app enumerates relationship labels in docs. Use `show_contact_relationships` to read the actual available relationship strings in the environment before filtering contacts by relationship.
[vc-00023] For current-date tasks, use `phone.get_current_date_and_time()` and verify the response field names are `date` and `time`. The date string may be a natural-language format (for example `%A, %B %d, %Y`) rather than ISO, so parse the returned format before deriving date windows.
[vc-00024] For Venmo mutation calls, verify success by inspecting the actual returned message text and any ids generated (for example `comment_id`) rather than assuming generic success strings.
[vc-00025] For relative-time tasks such as 'last 7 days' or 'including today', derive the cutoff from `phone.get_current_date_and_time()` and compare against transaction `created_at` dates; do not anchor the window to account `registered_at`, `last_logged_in`, or guessed calendar dates.
[vc-00029] When using `api_docs.show_api_descriptions`, expect a list of objects with `name` and `description`, not a flat string list. Copy the callable name exactly from that output before requesting docs or calling an API; if a guessed name like `show_notes`, `list_notes`, or `request_money` is not listed, do not try it.
[vc-00030] For Phone auth, verify the login identifier from docs: `phone.login` uses `username` = account `phone_number` (not email) and returns both `access_token` and `token_type`.
[vc-00031] For `phone.search_contacts`, include the required `access_token`; the endpoint is `GET /contacts` and contact objects use fields such as `first_name`, `last_name`, `email`, `phone_number`, `relationships`, `birthday`, `home_address`, `work_address`, and `created_at`.
[vc-00033] For `venmo.create_payment_request`, the recipient field is `user_email` and the note field is optional `description`; the call also supports optional `private` and returns both `message` and `payment_request_id` on success.
[vc-00034] For Simple Note, `login` follows the same auth pattern as Venmo: call `simple_note.login(username=<account email>, password=...)` and preserve both `access_token` and `token_type`. Do not substitute `email` for the parameter name `username`.
[vc-00035] For `simple_note.search_notes`, the endpoint is `GET /notes` and supports filters including `query`, `tags`, `pinned`, `dont_reorder_pinned`, `page_index`, `page_limit`, and `sort_by`. The response contains only note metadata (`note_id`, `title`, `tags`, `created_at`, `updated_at`, `pinned`), so fetch `show_note(note_id=..., access_token=...)` for content before parsing or acting.
[vc-00038] For file_system authentication, verify that `login` uses `username` and `password`, the path is `/auth/token`, and a successful response returns both `access_token` and `token_type`.
[vc-00039] For `file_system.show_directory`, do not assume pagination. The callable requires `access_token`, uses `directory_path` (default `/`) rather than `path`, supports optional `substring`, `entry_type` in `['all','files','directories']`, and `recursive`, and returns a flat list of path strings.
[vc-00040] For `file_system.show_file`, `file_path` is required. The response schema uses `file_id`, `path`, `content`, `created_at`, and `updated_at`; returned `content` may be a binary marker string rather than readable text, so use the timestamp metadata for photo/file organization tasks.
[vc-00041] For `file_system.move_file`, inspect and use the full contract before moving files: required fields are `source_file_path`, `destination_file_path`, and `access_token`; optional fields include `overwrite` and `retain_dates`; success returns both `message` and `destination_file_path`.
[vc-00044] When verifying filesystem results, normalize path expectations against actual API output: `show_directory` may return absolute `/home/...` paths even if you queried with `~/...`, and directory entries are represented with trailing `/`. Do not treat those formatting differences alone as evidence that moves or directory creation failed.
[vc-00045] For Spotify, do not assume every read endpoint requires authentication: `show_song(song_id=...)` takes only `song_id` and returns detailed metadata including `duration` and `artists`; omit `access_token` unless the docs for that specific endpoint require it.
[vc-00046] When computing Spotify playlist duration or inspecting playlist contents, use `show_playlist(... )` to get the nested `songs` list, whose entries use `id` for song ids. Then fetch `show_song(song_id=<song.id>)` for authoritative per-song fields such as `duration`; do not expect `show_playlist` to return `song_ids` or full song metadata in the same shape as `show_song`.
[vc-00047] For Spotify liked-playlist tasks, verify the dedicated `show_liked_playlists` contract rather than reusing playlist-library assumptions: its path is `/liked_playlists`, it supports optional `sort_by`, and each returned playlist record includes `liked_at` in addition to the usual playlist metadata.
[vc-00049] Before starting Spotify playback from search results, verify the selected playlist is semantically aligned with the requested activity as well as structurally sufficient. If the task asks for workout music, do not play a playlist chosen only because it has many songs or high ratings when its title/query match is weak or unrelated.
[vc-00050] For Spotify `show_current_song`, verify the actual success schema before using it in logic: it returns flat fields such as `song_id`, `title`, `album_id`, `album_title`, `duration`, `artists`, `played_seconds`, and `is_playing` rather than nested album objects or fields like `duration_secs`/`is_paused`.
[vc-00054] For Venmo `search_users`, an email query is not guaranteed to return only one exact user. Preserve the full candidate list, then choose recipients by exact `email` equality with the authoritative contact email; use fields like `friends_since` only as secondary confirmation, and do not create a payment request for a merely name-similar candidate.
## TROUBLESHOOTING AND PITFALLS:

[ts-00013] If local formatting code fails, consider sandbox import restrictions before debugging data logic. In particular, avoid relying on disallowed modules such as `io`; build strings directly or use only permitted standard-library pieces.
[ts-00037] In this sandbox, `print(...)` JSON-serializes many outputs. Avoid printing raw Python `set` objects or other non-JSON-serializable structures; convert them to lists/dicts/strings first to prevent TypeError during debugging.
[ts-00051] Do not assume `spotify.show_current_song(access_token=...)` will always succeed as a harmless state check. If the queue is empty, it can raise a 422 error (`{"message":"The song queue is empty."}`), so either handle that failure explicitly or start playback directly with `play_music(song_id=...|album_id=...|playlist_id=...)` when the task is to begin playing something.
