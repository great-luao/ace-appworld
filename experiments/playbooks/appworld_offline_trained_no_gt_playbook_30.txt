## APIs TO USE FOR SPECIFIC INFORMATION
[api-00004] You can use the "supervisor" app to get information about my accounts and use the "phone" app to get information about friends and family.

[api-00009] Spotify playlist-library schema shortcut: `apis.spotify.show_playlist_library(access_token=..., page_index=...)` returns a list of playlist objects, not just playlist IDs. Each playlist object already includes `playlist_id`, `title`, and `song_ids` (list of integers), so for tasks about songs inside your playlists, use `song_ids` directly and only call `show_playlist` if you specifically need extra playlist details not present in the library response.
[api-00010] Spotify song schema clarification: `apis.spotify.show_song(song_id=...)` does not require `access_token` and returns a song object including at least `title` and `like_count`. For 'most-liked song in playlists' tasks, compare songs by `like_count` and return the song `title`.
[api-00011] Spotify song-library schema clarification: `apis.spotify.show_song_library(access_token=..., page_index=...)` returns a paginated list of song objects that already include at least `song_id`, `title`, `album_id`, `album_title`, `duration`, `artists`, and `added_at`, but not `play_count`. For tasks like finding the least-/most-played song in the library, use `show_song_library` to enumerate all library songs, then call `apis.spotify.show_song(song_id=...)` for each `song_id` to read `play_count` and compare songs.
[api-00013] Spotify album-library schema clarification: `apis.spotify.show_album_library(access_token=..., page_index=...)` returns a paginated list of album objects, not just album IDs. Each album object already includes `album_id`, `title`, and `song_ids` (list of integers), so for tasks about songs inside your album library, collect `song_ids` directly from this response and only call `apis.spotify.show_album(album_id=...)` if you specifically need extra album details not present in the library response. Then use `apis.spotify.show_song(song_id=...)` (no `access_token` required) to read fields like `play_count` and compare songs.
[api-00016] Spotify liked-songs schema clarification: `apis.spotify.show_liked_songs(access_token=..., page_index=...)` returns a paginated list of song objects, not bare song IDs. Each item includes at least `song_id` and `title`, so for tasks involving songs you have liked, collect `song["song_id"]` directly from this response.
[api-00018] Spotify song-reviews schema clarification: `apis.spotify.show_song_reviews(song_id=..., user_email=..., page_index=...)` does not require `access_token`. It returns a paginated list of review objects including at least `song_review_id`, `rating`, `created_at`, and nested `user.email`. Even when `user_email` is supplied, still page until empty and re-check `review["user"]["email"]` before deciding whether to create or update a review.
[api-00020] Spotify song-review write API naming clarification: to create a new song rating/review, use `apis.spotify.review_song(song_id=..., rating=..., access_token=...)` — there is no `create_song_review` endpoint. To modify an existing review, use `apis.spotify.update_song_review(review_id=..., rating=..., access_token=...)`.
[api-00024] Phone contacts schema clarification: `apis.phone.search_contacts(access_token=..., relationship='roommate', page_index=...)` returns a paginated list of full contact objects, each including fields such as `email`, `phone_number`, and `relationships`. For roommate-related tasks, use this as the authoritative source and extract only the needed identifier(s), typically `contact['email']`.
[api-00025] Phone current-date schema clarification: `apis.phone.get_current_date_and_time()` returns an object like `{'date': 'Thursday, May 18, 2023', 'time': '...'}`. When comparing against ISO timestamps like `YYYY-MM-DDTHH:MM:SS`, convert the `date` string to ISO `YYYY-MM-DD` first; do not assume the string is already ISO-formatted.
[api-00026] Venmo social-feed schema clarification: `apis.venmo.show_social_feed(access_token=..., page_index=...)` returns paginated transaction objects that already include `transaction_id`, `created_at`, `sender.email`, `receiver.email`, and `like_count`. For tasks like liking today's roommate-related feed items, filter directly from this response by ISO-date prefix and roommate email membership, then call `apis.venmo.like_transaction(transaction_id=..., access_token=...)` on the matches.
[api-00027] Phone contacts schema extension for siblings: `apis.phone.search_contacts(access_token=..., relationship='sibling', page_index=...)` returns a paginated list of full contact objects, just like other relationship filters. For sibling-related tasks, treat this as the authoritative source and extract only the minimal identifier(s) needed, typically `contact['email']`.
[api-00032] Phone contacts schema extension for coworkers: `apis.phone.search_contacts(access_token=..., relationship='coworker', page_index=...)` is paginated and returns full contact objects including `email`, `phone_number`, and `relationships`. For coworker-related tasks, use this as the authoritative source and extract only `contact['email']` unless another field is explicitly needed.
[api-00034] Spotify export uniqueness rule: when a task asks for a unique list of songs gathered from multiple Spotify containers (for example song library + album library + playlists), aggregate all discovered songs into a map/set keyed by `song_id` first. Use `show_song_library` song objects directly when available, collect `song_ids` from album/playlist library responses, and call `apis.spotify.show_song(song_id=...)` only for IDs missing metadata. This avoids duplicate CSV rows for the same song appearing in multiple places.
[api-00035] File-system write schema clarification: `apis.file_system.create_file(file_path=..., access_token=..., content=..., overwrite=...)` is the correct endpoint to create or overwrite a file in this environment; do not guess a `write_file` endpoint. `apis.file_system.create_directory(directory_path=..., access_token=..., recursive=True)` creates parent directories as needed before writing backup files.
[api-00036] File-system read schema clarification: `apis.file_system.show_file(file_path=..., access_token=...)` returns a file object containing at least `content`. For backup-verification tasks, inspect this endpoint's doc before use, then validate the written artifact by checking expected header/content in `result['content']` before performing any destructive follow-up action.
[api-00042] Simple Note + Phone messaging schema clarification: `apis.simple_note.search_notes(access_token=..., query=..., page_index=...)` is paginated and returns note summary objects only (for example `note_id`, `title`, tags, timestamps) — not note contents. Use `apis.simple_note.show_note(note_id=..., access_token=...)` to retrieve the full note, including its `content` string. For message-driven tasks, `apis.phone.search_text_messages(access_token=..., query=..., phone_number=..., page_index=...)` returns paginated text-message objects that already include `sender`, `receiver`, `message`, and `sent_at`; use these fields directly to identify the correct conversation and reply target.
[api-00043] Phone text-message write schema clarification: `apis.phone.send_text_message(phone_number=..., message=..., access_token=...)` is the correct endpoint to reply by SMS. It returns a success object including at least `message` and `text_message_id`.
[api-00046] Venmo received-payments schema clarification: for tasks specifically about payments you received, prefer `apis.venmo.show_transactions(...)` over `show_social_feed` because it is the direct source of truth for your own transaction history and supports server-side filtering by `direction='received'`, `min_created_at`, and `max_created_at`. Its paginated transaction objects include at least `transaction_id`, `created_at`, `sender.email`, `receiver.email`, `like_count`, and `comment_count`, which is enough to filter coworker-sent received payments without extra `show_transaction` calls.
[api-00047] Venmo transaction-write schema clarification: to add a payment comment, use `apis.venmo.create_transaction_comment(transaction_id=..., comment=..., access_token=...)`; to like the payment, use `apis.venmo.like_transaction(transaction_id=..., access_token=...)`. `create_transaction_comment` returns a success object including at least `message` and `comment_id`, while `like_transaction` returns a success object including at least `message`.
[api-00052] Spotify recommendations schema clarification: `apis.spotify.show_recommendations(access_token=..., page_index=..., page_limit=...)` returns a paginated list of song objects, each including at least `song_id`, `title`, `album_id`, `album_title`, `duration`, and `artists`, where `artists` is a list of artist objects with fields like `id` and `name`. For tasks asking who is 'most recommended' on Spotify, use this endpoint as the primary source and aggregate from the nested artist names in the returned recommended songs.
[api-00059] Spotify newest/oldest-release schema clarification: `apis.spotify.show_song_library(access_token=..., page_index=...)` is useful for enumerating saved songs, but its song objects do not include `release_date`. For tasks asking for the newest/oldest song across song, album, and playlist libraries, treat `apis.spotify.show_song(song_id=...)` as the source of truth for each candidate song's `title` and `release_date` (no `access_token` required), while using album/playlist library responses only to enumerate additional `song_ids`.
[api-00062] Venmo payment-request schema clarification: to request money from someone, use `apis.venmo.create_payment_request(user_email=..., amount=..., access_token=..., description=..., private=...)` — not a guessed endpoint such as `request_money`. It returns a success object including at least `message` and `payment_request_id`.
[api-00068] Venmo sent-payment-requests schema clarification: `apis.venmo.show_sent_payment_requests(access_token=..., status='pending', page_index=...)` returns a paginated list of sent payment-request objects including at least `payment_request_id`, `amount`, `description`, `created_at`, `receiver.email`, and `sender.email`. Use this endpoint to detect already-pending outbound requests before creating new ones.

## COMMON MISTAKES AND CORRECT STRATEGIES

[cms-00014] When parsing structured note/text content, be careful with whitespace normalization before prefix checks. If you call `line.strip()`, a subsequent check like `line.startswith(' - ')` will always fail because leading spaces were removed. Either test the raw line before stripping, or normalize first and then check the matching normalized prefix (for example, `line.startswith('-')`).
[cms-00017] When using review-list endpoints with filters (for example `apis.spotify.show_song_reviews(song_id=..., user_email=...)`), do not assume the filter guarantees a single result or a single page. Page until empty, then select the correct matching review record by `user.email`/`song_review_id`; if multiple self-reviews are present, update the appropriate existing review rather than blindly taking the first item from the first page.
[cms-00022] When a task only needs identifiers from contact data (for example roommate emails), do not print or store full contact records in output. Extract only the minimal fields needed (such as `email`) and keep private fields like phone numbers, birthdays, and addresses out of logs/debug prints.
[cms-00023] Do not hard-code date literals seen in the environment when filtering time-based records. If an API returns a human-readable current date but target records use ISO timestamps, parse/convert the returned date into the comparable format programmatically before filtering.
[cms-00028] Before performing repeated write actions from a filtered list of records (for example, liking Venmo transactions), deduplicate the target IDs first and avoid printing raw IDs unless the task explicitly requires them. This reduces redundant writes and unnecessary exposure of private activity details in logs/debug output.
[cms-00033] For Venmo social-feed tasks phrased as transactions 'involving' a relationship group (for example coworkers), treat a feed item as matching if either `transaction['sender']['email']` or `transaction['receiver']['email']` is in the target email set. Because `show_social_feed` already includes both parties' emails and `created_at`, do not make extra `show_transaction` calls just to re-check involvement.
[cms-00037] In export tasks that combine backup with an irreversible action, do not treat a successful write call alone as sufficient. First verify the backup artifact with a read endpoint and confirm expected content shape (for example exact CSV header), then perform the destructive action such as account deletion.
[cms-00038] For simple export/backup formats (especially CSV), do not assume Python helper modules are usable in this environment. Avoid unnecessary dependencies on modules like `io`/`csv` when a manual string builder is straightforward; construct the content directly, including proper CSV escaping for quotes, commas, and newlines, to prevent preventable execution failures.
[cms-00040] For CSV export tasks where the user specifies exact header names (for example `Title` and `Artists`), prefer writing the header row exactly as `Title,Artists` unless quoting is explicitly needed. Quoted headers like `"Title","Artists"` are valid CSV, but exact literal headers are safer when an evaluator may compare the first line strictly.
[cms-00041] For phone/message tasks involving a person name search (for example, replying to 'Christopher'), do not select `search_contacts(query=...)` results by position such as the first match. If multiple contacts share the name, treat the actual inbound/outbound message thread as the source of truth: identify the relevant text message first, derive the exact phone number from that message object, and only use contacts to confirm metadata if needed.
[cms-00044] For phone/Simple Note tasks, avoid printing full note contents or full text-message bodies during debugging. Use privacy-minimal checks instead, such as note titles/IDs, message counts, timestamps, or the extracted final reply payload.
[cms-00045] For message-driven lookup tasks where the inbound text specifies a qualifier (for example a director, artist, genre, or other entity), do not infer matches with loose token-overlap heuristics like checking whether any word from a metadata field appears in the request text. First extract the exact requested qualifier from the message, normalize it, then filter structured records by exact or otherwise high-confidence normalized match on the relevant field before sending the reply.
[cms-00048] For Venmo tasks phrased as payments you 'received' from a relationship group, do not use broad involvement heuristics alone (such as feed membership where either side matches). Use `show_transactions(direction='received', ...)` as the authoritative source, then filter by `sender.email` in the relationship-email set and, when the account email is already given in the task, `receiver.email` matching that user email.
[cms-00049] When a Venmo task requires adding the same comment to many transactions, deduplicating transaction IDs is necessary but not always sufficient for robustness. If there is any chance the workflow may be rerun and duplicate comments would be undesirable, optionally inspect existing transaction comments before creating a new one so you do not add repeated identical comments unnecessarily.
[cms-00050] For Venmo multi-write tasks that add a fixed comment and like matching transactions, treat rerun safety as part of correctness, not just a nice-to-have. After filtering and deduplicating target `transaction_id`s, check whether the exact comment already exists on each transaction if the environment exposes a comments-reading endpoint; skip creating duplicate identical comments, then proceed with the like action.
[cms-00051] For date-window tasks phrased like 'last N days (including today)', use an inclusive window ending on today's date: set the lower bound to `today - (N - 1) days`, not `today - N days`. Example: 'last 10 days (including today)' means `min_created_at = today - 9 days` and `max_created_at = today`.
[cms-00053] For recommendation/list-aggregation tasks (for example 'most recommended artist'), choosing the correct endpoint is not enough; define the aggregation rule explicitly before answering. Decide whether to count artist appearances across all returned items, whether duplicate rows should be deduplicated first (for example by `song_id`), and do not rely on bare `max()` if multiple entities can tie for first place.
[cms-00056] In this REPL environment, do not rely on notebook-style implicit expression echo for verification. A final bare expression like `counter.most_common(5)` may return only a generic success message. When you need to inspect an intermediate result, assign it to a variable and `print(...)` the specific non-sensitive value you need.
[cms-00061] For Spotify oldest/newest song tasks across multiple libraries, avoid re-fetching song metadata during tie-checks. After deduplicating candidate `song_id`s, call `apis.spotify.show_song(song_id=...)` once per unique song, cache each returned song object in a dictionary keyed by `song_id`, compute the extreme `release_date` from that cache, and reuse the same cached objects to verify whether the minimum/maximum date is unique.
[cms-00063] For coworker-payment tasks that parse attendee names from a note, do not resolve people by first name alone unless that first name is unique among the relevant coworker contacts. Build the mapping primarily from exact normalized full names; only allow first-name fallback after checking uniqueness, otherwise require stronger disambiguation before writing payments/requests.
[cms-00064] For Venmo request-creation tasks, rerun safety should compare the full intended request identity, not just the receiver. When excluding already-pending requests, match on at least `receiver.email` + `description`, and preferably `amount` as well, so you do not skip or duplicate requests incorrectly when multiple requests to the same person exist.
[cms-00066] For Venmo payment/request tasks driven by attendee names from a note, do not treat `phone.search_contacts(query=<first_name>)` as sufficient identity proof. If the note only gives first names, search contacts only as a candidate source, then require that each name resolve to exactly one unique contact after deduplication by a stable key such as `email`/`contact_id`; otherwise do not write until you have stronger disambiguation. This rule is not limited to coworker tasks.
[cms-00069] For Venmo reimbursement tasks where you must infer who has already paid from received transactions, do not treat `(sender.email, amount)` alone as sufficient evidence. After date-window filtering, match the strongest available payment identity — at minimum `sender.email + amount + exact description`, and when the task already provides your account identity also require `receiver.email` to match you — so unrelated same-amount payments are not mistaken for reimbursements.
[cms-00071] For `apis.phone.get_current_date_and_time()['date']`, do not extract the year or month by ad-hoc string splitting/token positions. The returned value is a human-readable string like `Thursday, May 18, 2023`; parse it with the documented format (for example `datetime.strptime(date_str, '%A, %B %d, %Y')`) before building ISO bounds such as `YYYY-02-01` or `YYYY-01-01`. This avoids bugs like using the day-of-month as the year.
