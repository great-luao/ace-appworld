## APIs TO USE FOR SPECIFIC INFORMATION
[api-00004] You can use the "supervisor" app to get information about my accounts and use the "phone" app to get information about friends and family.

[api-00009] For account credentials, `supervisor.show_account_passwords()` returns a direct list of objects with `account_name` and `password` only. Find the needed app password by filtering that list; do not assume a wrapped `passwords` field or extra username field.
[api-00012] For questions about songs appearing in the user's playlists, start with `spotify.show_playlist_library`: its playlist objects include `playlist_id`, privacy/owner metadata, and `song_ids`. You can aggregate unique `song_ids` directly from this endpoint without calling `show_playlist` unless track-level playlist details are specifically needed.
[api-00013] For Spotify song details, use `spotify.show_song(song_id=...)`. The response includes the key popularity field `like_count` and uses schema fields such as `album_title`, `duration`, and a nested `artists` list; do not assume flatter fields like `artist` or `duration_secs`.
[api-00014] For Spotify song-library questions about metrics such as least/most played, do not expect `show_song_library` to contain `play_count`. Use `show_song_library` only to enumerate library songs (it returns fields like `song_id`, `title`, `album_id`, `album_title`, `duration`, `artists`, `added_at`), then call `spotify.show_song(song_id=...)` for each library song to obtain `play_count` and other detailed song metrics before comparing or ranking.
[api-00015] For Spotify album-library tasks, use `spotify.show_album_library` as the source of album membership: each returned album includes `song_ids` directly, and the endpoint supports both `page_index` and `page_limit`. Paginate through all album-library pages, collect `song_ids` from the album objects, and only then call `spotify.show_song(song_id=...)` for per-song metrics such as `play_count`.
[api-00016] For Spotify liked-song tasks, `spotify.show_liked_songs` is paginated and requires `access_token`; it also supports `page_limit` and `sort_by`. Each item includes at least `song_id`, `title`, `album_id`, `album_title`, `duration`, `artists` (with `id` and `name`), and `liked_at`.
[api-00017] For Spotify playlist-library tasks, `spotify.show_playlist_library` uses path `/library/playlists` and can filter by optional `is_public`. Returned playlist objects include nested `owner` data plus `song_ids`; do not assume flat owner fields.
[api-00018] For Spotify song-rating/review tasks, first use `spotify.show_song_reviews(song_id=..., user_email=...)` to find the current user's existing review for each song. The endpoint is paginated and supports filters such as `user_email`, `query`, `min_rating`, `max_rating`, and `sort_by`.
[api-00019] Spotify review write APIs are asymmetric: `spotify.review_song(song_id=..., rating=..., access_token=..., title?, text?)` creates a review and returns only `{message, song_review_id}`, while `spotify.update_song_review(review_id=..., access_token=..., rating?, title?, text?)` updates by `review_id` and returns only `{message}`. Do not expect create/update calls to return the full review object.
[api-00022] To determine per-song user-private state in Spotify, use `spotify.show_song_privates(song_id=..., access_token=...)`. Its path is `/songs/{song_id}/privates`, and the response includes `liked`, `reviewed`, `in_song_library`, and `downloaded`. For tasks about songs in the library that have or have not been liked/reviewed/downloaded, fetch the library song ids first, then query this endpoint per song.
[api-00023] To find whether the current user already reviewed a Spotify song, do not use `show_song_review` for discovery. `show_song_review(review_id=...)` only retrieves a known review by id. Instead, call `spotify.show_song_reviews(song_id=..., user_email=...)` and use the returned `song_review_id` for any later `update_song_review` call.
[api-00025] For relationship-based identity tasks involving roommates, friends, or relatives, use the Phone app as the authoritative source. `phone.search_contacts` requires `access_token`, supports filters such as `relationship`, `query`, `page_index`, and `page_limit`, and returns contact records with fields including `contact_id`, `first_name`, `last_name`, `email`, `phone_number`, and `relationships`.
[api-00027] For Venmo social-feed tasks, use `venmo.show_social_feed(access_token=..., page_index?, page_limit?)`. The endpoint path is `/social_feed`, default `page_limit` is 5, and each item includes `transaction_id`, `amount`, `description`, `created_at`, `updated_at`, `private`, `like_count`, `comment_count`, and nested `sender`/`receiver` objects with `name` and `email`.
[api-00028] For Venmo transaction-like actions, `venmo.like_transaction(transaction_id=..., access_token=...)` uses path `/transactions/{transaction_id}/likes` and returns a success object containing only a `message` string. Do not assume a boolean `liked` field in the response.
[api-00029] For Phone relationship-based tasks, first call `phone.show_contact_relationships(access_token=...)` to discover the exact available relationship labels before filtering contacts. Do not assume labels like `sister`, `spouse`, or other intuitive variants exist; use the returned values such as `sibling` when present.
[api-00031] For tasks involving recency like 'today', 'yesterday', or similar windows, use `phone.get_current_date_and_time()` as the source of truth. Its path is `/date_time`, it takes no parameters, and returns `{date, time}` as human-readable strings, so parse `date` before comparing it to ISO timestamps from other apps such as Venmo `created_at`.
[api-00032] For file writes, do not invent filesystem API names from standard library intuition. In the `file_system` app, inspect docs and use APIs such as `create_directory(directory_path, access_token, recursive?)`, `create_file(file_path, access_token, content?, overwrite?)`, `show_file`, and `update_file`; paths may be absolute or `~/`-relative.
[api-00034] For Spotify account-termination tasks, use `spotify.delete_account(access_token=...)`; do not invent names like `terminate_account`. The endpoint deletes the account and returns only a success/failure message.
[api-00035] Use `spotify.show_playlist(playlist_id=..., access_token=...)` only when you need playlist-level details beyond `show_playlist_library`. Its `songs` entries are lightweight objects shaped like `{id, title, artist_ids}` rather than full song records, so fetch each missing song with `spotify.show_song(song_id=...)` when you need artist names or other song metadata.
[api-00038] For Simple Note tasks, use `simple_note.search_notes(access_token=..., query?, tags?, pinned?, dont_reorder_pinned?, page_index?, page_limit?, sort_by?)` only to find candidate notes; it does not return note content. [api-00039] For Phone text-message tasks, `phone.show_text_message_window` requires both `phone_number` and `access_token` and uses optional `min_datetime`, `max_datetime`, `pagination_order`, `page_index`, and `page_limit`. `phone.send_text_message` requires the parameter name `message` (not `content`) plus `phone_number` and `access_token`.

[api-00046] For Venmo tasks involving your own payments sent or received, use `venmo.show_transactions` rather than `show_social_feed`. [api-00049] For `phone.search_contacts`, do not assume a flat contact schema. [api-00053] For Spotify questions scoped across the user's song, album, and playlist libraries, treat those libraries as three separate sources of song membership. [api-00055] For Spotify library endpoints `show_song_library`, `show_album_library`, and `show_playlist_library`, remember that results are paginated with default `page_limit=5` and continue beyond the first page. ## COMMON MISTAKES AND CORRECT STRATEGIES

[cms-00020] When a task asks to rate items that may already have ratings, do not blindly call the create-review endpoint for every target item. Fetch the user's existing reviews first, then create reviews for unrated items and update reviews whose current rating is below the requested value.
[cms-00021] When a task asks to set song ratings to a specific target value, compare each existing review's current `rating` against that target and update any review whose rating differs from the requested value, including decreasing higher ratings as well as increasing lower ratings. Only create a new review when the user has no existing review for that item.
[cms-00030] When collecting contacts via `phone.search_contacts` across multiple relationship filters (for example `sibling` and `brother`), expect overlap: the same person can be returned for more than one relationship label. Deduplicate contacts by a stable key such as `contact_id` or email before counting, matching transactions, or taking write actions.
[cms-00033] In this environment, some Python modules are blocked even for local formatting work; avoid `io` for CSV construction. When exporting simple CSVs, build the string directly (including header row and proper quote escaping) or use only allowed modules.
[cms-00040] Do not identify a contact from `phone.search_contacts(query=...)` by assuming the query returns only matching names. Broad queries can return many unrelated contacts across multiple pages; inspect the returned records and verify the intended person using exact name or other corroborating fields before acting.
[cms-00041] When asked to reply to an existing conversation using information from another app, first inspect the recent message history to recover the exact request details or constraints, then extract only the matching subset from the source data. Do not assume the task prompt alone tells you which portion of a note or list should be sent.
[cms-00043] When replying to a person based on prior texts, a broad `phone.search_text_messages(query=...)` may return many messages across contacts and dates. Do not act on the first apparent hit; inspect the nested `sender`/`receiver` fields, isolate the correct contact phone number, and choose the latest relevant inbound request before composing the reply.
[cms-00044] For Simple Note lookup tasks, do not assume `search_notes(query=...)` returns only tightly matching notes or that the first hit is the right one. Search results can include loosely related or unrelated notes and may be reordered by pinned/query behavior, so verify the intended note using metadata such as title/tags and then confirm by calling `show_note` before extracting content.
[cms-00045] When replying from a note that contains a formatted list plus per-item metadata, parse the note content and extract only the entries satisfying the contact's latest request constraint (for example a specific director), then send just the requested item names in the requested output format rather than copying the whole note text.
[cms-00048] For Venmo tasks scoped to a recent time window such as 'last 7 days' or 'including today', do not derive the cutoff from account metadata like `registered_at`/`last_logged_in` or from guessed dates. First get the current date via `phone.get_current_date_and_time()`, compute the cutoff from that date, and then filter transactions by `created_at`.
[cms-00051] For Spotify recommendation questions phrased at the artist level (for example, least/most recommended artist), fetch all pages from `show_recommendations` first, then count occurrences across every song's nested `artists` list before ranking. Do not answer from a partial page or from only one artist field per song.
[cms-00056] When a task requires sending Venmo money or payment requests to people named in notes/messages, do not trust a single `venmo.search_users(query=...)` or `phone.search_contacts(query=...)` hit. These searches can return many unrelated candidates across pages. Resolve the person by cross-checking exact first/last name plus corroborating fields from another authoritative source—typically matching email from prior Venmo transactions or Phone contacts—before taking the write action.
[cms-00058] When sending Venmo payment requests for people listed in a note, first resolve each person in Phone contacts to get the authoritative email, then match that email against Venmo search results; use `friends_since` as supporting evidence, not as a substitute for exact email matching, before taking the write action.
[cms-00059] Do not invent note-listing or Venmo request endpoint names from intuition.
