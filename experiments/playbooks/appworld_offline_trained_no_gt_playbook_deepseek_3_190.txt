## STRATEGIES AND HARD RULES
[shr-00001] Make sure to end code blocks with ``` followed by a newline(\n).
[shr-00005] Always look at API specifications (using apis.api_docs.show_api_doc) before calling an API.
[shr-00006] Write small chunks of code and only one chunk of code in every step. Make sure everything is working correctly before making any irreversible change.

[shr-00015] Do not print or otherwise expose secrets retrieved from supervisor (passwords, tokens, verification codes). Fetch credentials into variables and use them directly in subsequent API calls without echoing them to output unless the task explicitly requires revealing them.
[shr-00019] Avoid unnecessary private-data calls and output. If the task already gives the needed identity/account context, do not call endpoints like `show_account` just to reconfirm it, and do not print private account details unless they are required to solve the task.
[shr-00031] When you need multiple account credentials, call `apis.supervisor.show_account_passwords()` only once, store the returned list in a variable, and extract all needed passwords from that cached result. Avoid repeated credential-fetch calls and never print the returned password records.
[shr-00039] Keep debug output privacy-minimal in credentialed workflows: never print password records, access tokens, or raw login response objects, and do not print full backup contents during verification. Prefer silent variable assignment and, if you must sanity-check progress, print only non-sensitive summaries such as boolean login success, counts, or header-only validation.

## COMMON MISTAKES AND CORRECT STRATEGIES

[cms-00014] When parsing structured note/text content, be careful with whitespace normalization before prefix checks. If you call `line.strip()`, a subsequent check like `line.startswith(' - ')` will always fail because leading spaces were removed. Either test the raw line before stripping, or normalize first and then check the matching normalized prefix (for example, `line.startswith('-')`).
[cms-00017] When using review-list endpoints with filters (for example `apis.spotify.show_song_reviews(song_id=..., user_email=...)`), do not assume the filter guarantees a single result or a single page. Page until empty, then select the correct matching review record by `user.email`/`song_review_id`; if multiple self-reviews are present, update the appropriate existing review rather than blindly taking the first item from the first page.
[cms-00022] When a task only needs identifiers from contact data (for example roommate emails), do not print or store full contact records in output. Extract only the minimal fields needed (such as `email`) and keep private fields like phone numbers, birthdays, and addresses out of logs/debug prints.
[cms-00023] Do not hard-code date literals seen in the environment when filtering time-based records. If an API returns a human-readable current date but target records use ISO timestamps, parse/convert the returned date into the comparable format programmatically before filtering.
[cms-00028] Before performing repeated write actions from a filtered list of records (for example, liking Venmo transactions), deduplicate the target IDs first and avoid printing raw IDs unless the task explicitly requires them. This reduces redundant writes and unnecessary exposure of private activity details in logs/debug output.
[cms-00033] For Venmo social-feed tasks phrased as transactions 'involving' a relationship group (for example coworkers), treat a feed item as matching if either `transaction['sender']['email']` or `transaction['receiver']['email']` is in the target email set. Because `show_social_feed` already includes both parties' emails and `created_at`, do not make extra `show_transaction` calls just to re-check involvement.
[cms-00037] In export tasks that combine backup with an irreversible action, do not treat a successful write call alone as sufficient. First verify the backup artifact with a read endpoint and confirm expected content shape (for example exact CSV header), then perform the destructive action such as account deletion.
[cms-00038] For simple export/backup formats (especially CSV), do not assume Python helper modules are usable in this environment. Avoid unnecessary dependencies on modules like `io`/`csv` when a manual string builder is straightforward; construct the content directly, including proper CSV escaping for quotes, commas, and newlines, to prevent preventable execution failures.
[cms-00040] For CSV export tasks where the user specifies exact header names (for example `Title` and `Artists`), prefer writing the header row exactly as `Title,Artists` unless quoting is explicitly needed. Quoted headers like `"Title","Artists"` are valid CSV, but exact literal headers are safer when an evaluator may compare the first line strictly.
[cms-00041] For phone/message tasks involving a person name search (for example, replying to 'Christopher'), do not select `search_contacts(query=...)` results by position such as the first match. If multiple contacts share the name, treat the actual inbound/outbound message thread as the source of truth: identify the relevant text message first, derive the exact phone number from that message object, and only use contacts to confirm metadata if needed.
[cms-00044] For phone/Simple Note tasks, avoid printing full note contents or full text-message bodies during debugging. Use privacy-minimal checks instead, such as note titles/IDs, message counts, timestamps, or the extracted final reply payload.
[cms-00045] For message-driven lookup tasks where the inbound text specifies a qualifier (for example a director, artist, genre, or other entity), do not infer matches with loose token-overlap heuristics like checking whether any word from a metadata field appears in the request text. First extract the exact requested qualifier from the message, normalize it, then filter structured records by exact or otherwise high-confidence normalized match on the relevant field before sending the reply.
[cms-00048] For Venmo tasks phrased as payments you 'received' from a relationship group, do not use broad involvement heuristics alone (such as feed membership where either side matches). Use `show_transactions(direction='received', ...)` as the authoritative source, then filter by `sender.email` in the relationship-email set and, when the account email is already given in the task, `receiver.email` matching that user email.
[cms-00049] When a Venmo task requires adding the same comment to many transactions, deduplicating transaction IDs is necessary but not always sufficient for robustness. If there is any chance the workflow may be rerun and duplicate comments would be undesirable, optionally inspect existing transaction comments before creating a new one so you do not add repeated identical comments unnecessarily.
[cms-00050] For Venmo multi-write tasks that add a fixed comment and like matching transactions, treat rerun safety as part of correctness, not just a nice-to-have. After filtering and deduplicating target `transaction_id`s, check whether the exact comment already exists on each transaction if the environment exposes a comments-reading endpoint; skip creating duplicate identical comments, then proceed with the like action.
